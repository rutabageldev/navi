# ADR-001: Navi Repo Scope and Personal Enablement Charter

## Status
Accepted

## Date
2026-04-06

## Context

A new repository is being created to house a personal intelligence and
enablement system. Prior to this, no such system existed. The author
maintains two other active repositories -- Foundation (shared
infrastructure) and ruby-core (home automation) -- and has established
a philosophy of grouping repositories by primary function rather than
by technology or team.

A product strategy document has been produced that defines the vision,
principles, feature horizon, and out-of-scope boundaries for this
system. That document is the authoritative product reference. This ADR
records the architectural decision to create a new repository, defines
its scope, and establishes the charter that governs what belongs here
versus in adjacent repositories.

The system has been named Navi. The name is intentional -- it evokes a
guide that surfaces what matters, anticipates needs, and stays close
without getting in the way.

## Decision

A new repository named `navi` is created to house all personal
enablement and intelligence features. Its scope is defined as follows:

**Navi owns:**
- Intelligence delivery features (news digest, trend analysis, research
  briefs, morning brief)
- Relationship intelligence and Rolodex
- Professional profile and career context
- Communication triage and personal email awareness
- Inbound interaction handling (SMS via Twilio)
- Outbound delivery (email via Resend, SMS via Twilio, HA notifications)
- All Postgres schemas prefixed with `navi_`
- All Vault paths prefixed with `secret/data/navi/`
- All NATS subjects prefixed with `navi.>`

**Navi does not own:**
- Home automation (ruby-core)
- Shared infrastructure -- Postgres, NATS, Vault, Traefik (Foundation)
- Infrastructure context and network state (ruby-mesh, planned)
- Work systems of any kind -- corporate email, calendar, or internal
  tooling are permanently out of scope

**Navi consumes but does not control:**
- NATS JetStream from ruby-core as the event bus
- Postgres from Foundation as the data store
- Home Assistant from ruby-core for notification delivery
- ruby-core event streams for Ada and home context (read-only)

## Repo Structure

```
navi/
├── CLAUDE.md                  # Agent prompt format and conventions
├── README.md
├── STRATEGY.md                # Product strategy -- the why behind all decisions
├── docs/
│   └── adr/                   # All ADRs live here
├── services/
│   └── digest/                # Daily intelligence service (v1)
├── docker-compose.yml
├── docker-compose.staging.yml
├── docker-compose.dev.yml
└── Makefile
```

Additional services are added under `services/` as features are built.
Each service is independently deployable with its own Dockerfile and
owns a distinct set of NATS subjects and Postgres tables.

## Consequences

**Positive:**
- Clear scope boundary means future feature decisions have a documented
  home. "Does this belong in navi?" is answerable by asking whether it
  serves personal enablement for the primary user.
- Navi can evolve independently of home automation and infrastructure
  concerns.
- The strategy document and ADRs live in the repo, so architectural
  intent is preserved alongside the code.

**Negative / tradeoffs:**
- Navi takes a runtime dependency on ruby-core's NATS instance. If
  ruby-core is unavailable, Navi's internal event bus is unavailable.
  This is accepted as a practical tradeoff for a single-node homelab.
  The NATS migration to Foundation is flagged as future work in
  ADR-002.
- Navi takes a runtime dependency on Foundation's Postgres instance.
  This is consistent with ruby-core's existing dependency and is
  accepted.

## Alternatives Considered

**Fold Navi into Foundation**
Rejected. Foundation is infrastructure -- it supports other services
but delivers no direct user value on its own. Navi delivers direct
personal value and will grow into a significant feature surface. It
deserves its own repository and charter.

**Fold Navi into ruby-core**
Rejected. ruby-core is home automation. Navi is personal intelligence.
These are distinct problem domains with different data models,
dependencies, and cadences. Conflating them would compromise the scope
discipline that makes both repositories maintainable.

**Create a separate repo per Navi feature**
Rejected. Features within Navi share a data model, a delivery
abstraction, and a common identity. Splitting them into separate repos
would create coordination overhead with no meaningful benefit at this
scale.
