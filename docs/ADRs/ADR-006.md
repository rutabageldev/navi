# ADR-006: Delivery and Inbound Channel Abstraction

## Status
Accepted

## Date
2026-04-06

## Context

Navi communicates with the user in two directions: outbound delivery
of intelligence (digests, briefs, alerts, notifications) and inbound
receipt of instructions (directed actions via SMS). These channels
must be designed as an abstraction rather than hardcoded integrations,
because the delivery surface will grow over time -- audio output for
the commute, HA notifications for simple actions, and potentially
additional channels as new features are built.

The wrong design here is a delivery module that knows it is sending
a news digest via email. The right design is a delivery module that
knows how to render and send a structured payload to a configured
channel, agnostic to the content it carries.

Current channel decisions (documented in the product strategy and
confirmed prior to this ADR):
- Outbound primary: email via Resend
- Inbound and outbound conversational: SMS via Twilio
- Outbound simple actions: Home Assistant companion app notifications
- Future outbound: audio/TTS for commute consumption

## Decision

### Outbound Delivery Abstraction

All outbound delivery is handled by a `delivery` package within each
service that requires it (initially the digest service). The delivery
package defines a `Payload` interface:

```go
type Payload interface {
    Subject() string
    Sections() []Section
    Metadata() map[string]string
}

type Section struct {
    Title   string
    Body    string
    Links   []Link
}
```

A `Dispatcher` accepts a `Payload` and routes it to one or more
configured `Channel` implementations. Adding a new output format
(audio, HA push) is a new `Channel` implementation -- nothing else
changes.

```go
type Channel interface {
    Name() string
    Send(ctx context.Context, p Payload) error
}
```

### Email Channel (Resend)

The email channel renders the `Payload` as a structured HTML email
and sends it via the Resend API.

The HTML template follows a consistent structure:
- Header: "Hey, listen!" in the Navi accent color, date, digest type
- Sections rendered in order with clear visual separation
- Each article entry: headline, source badge, summary, citation link
- Footer: digest metadata, unsubscribe note (for deliverability)

The template is designed from day one to accommodate sections beyond
news articles -- calendar context, Ada summary, todo nudges -- so the
unified morning brief (Next horizon) requires only new section types,
not a template redesign.

Resend API credentials are retrieved from Vault at
`secret/data/navi/{env}/resend`. In staging, the sandbox key renders
emails without delivering them, providing full template validation
without sending real mail.

```
Vault path:       secret/data/navi/{env}/resend
Required keys:    api_key, from_address, to_address
```

### SMS Channel (Twilio)

The SMS channel handles both outbound delivery and inbound instruction
handling. These are related but distinct responsibilities.

**Outbound SMS**
Short-form messages and conversational responses sent to the user's
registered mobile number. SMS output is deliberately terse -- it is
not a digest format, it is a notification and response surface.

Twilio credentials are retrieved from Vault at
`secret/data/navi/{env}/twilio`. In staging, Twilio test credentials
simulate sends without delivering real messages or incurring cost.

```
Vault path:       secret/data/navi/{env}/twilio
Required keys:    account_sid, auth_token, from_number, to_number
```

**Inbound SMS (Twilio webhook)**
Navi exposes a webhook endpoint that Twilio calls on inbound SMS
from the registered number. The endpoint is exposed via Traefik on
the standard Navi ingress.

```
POST /webhooks/twilio/inbound
```

The inbound handler:
1. Validates the Twilio request signature to reject spoofed requests
2. Publishes the message body to `navi.{env}.sms.inbound`
3. Returns a 200 immediately (Twilio requires a fast response)

A separate `intent` component subscribes to `navi.{env}.sms.inbound`
and routes messages to the appropriate handler based on parsed intent.
In v1, intent parsing is handled by a Claude API call with a
structured prompt that returns a JSON action object:

```json
{
  "intent": "calendar_create",
  "params": {
    "title": "Interior designer meeting",
    "date": "next Tuesday",
    "time": null
  }
}
```

Supported intents in v1:
- `calendar_create` -- create a calendar event (via Google Calendar API)
- `profile_update` -- add an entry to the professional portfolio
- `rolodex_add` -- add a person to the Rolodex
- `unknown` -- Navi responds asking for clarification

All intent handlers follow the principle from ADR-001: Navi acts on
explicit direction only. Every handled intent sends a confirmation SMS
back to the user before taking action.

### Home Assistant Notification Channel

Navi sends actionable push notifications to the Home Assistant
companion app for simple binary interactions that do not require a
typed response. These are delivered via the HA REST API.

Use cases:
- "Your meeting with [person] is tomorrow morning. Generate prep brief?" [Yes / Skip]
- "Navi v1.2.0 deployed to staging. Promote to prod?" [Promote / Hold]
- Relationship nudge acknowledgements

HA notifications are a complement to the email and SMS channels, not
a replacement. They are used only when a single tap is the natural
response -- never for content that requires reading or typing.

```
Vault path:       secret/data/navi/{env}/homeassistant
Required keys:    url, token
```

### Future: Audio Channel

The audio channel is not built in v1 but is a first-class future
output format. The `Payload` abstraction is designed to accommodate
it without modification: an audio channel implementation would render
each `Section` as a natural language script and pass it to a TTS
provider (ElevenLabs or similar).

The implication for the email template design is that sections must
carry semantic meaning, not just HTML structure. A section's `Body`
field must be readable as prose, not as visual layout. This constraint
is applied from day one so audio output does not require a content
refactor.

### Channel Configuration

Channels are configured per environment via Vault. The dispatcher
reads a channel manifest at startup that defines which channels are
active and what their routing rules are.

In v1 the routing is simple:
- digest.ready events -> email channel
- sms.outbound events -> SMS channel
- ha.notify events -> HA channel

As new features are added, new routing rules are registered. The
dispatcher does not need to know about feature-specific logic -- it
routes payloads to channels based on event type.

## Consequences

**Positive:**
- The Payload/Channel abstraction means the audio output format
  (Later horizon) is a new implementation, not a redesign.
- The unified morning brief (Next horizon) is a new Payload
  composition, not a new delivery system.
- The inbound webhook and intent model provides a clean, extensible
  pattern for adding new directed actions without modifying the
  webhook handler.
- Twilio test credentials in staging mean no real SMS messages are
  sent during smoke tests.

**Negative / tradeoffs:**
- The inbound webhook requires a publicly reachable endpoint or a
  Twilio-compatible tunnel. Traefik handles TLS termination and
  routing; the endpoint is protected by Twilio signature validation.
  This is the only Navi endpoint that accepts inbound connections
  from the public internet.
- Intent parsing via Claude API adds latency to SMS responses.
  For conversational interactions, a response time of 2-5 seconds
  is acceptable. If latency becomes problematic, a lightweight
  keyword-based pre-filter can reduce Claude API calls for common
  intents.

## Alternatives Considered

**Hardcode email delivery in the digest service**
Rejected. The digest service is the first feature, not the last.
Hardcoding email means retrofitting an abstraction when the morning
brief needs to combine digest content with calendar context. The
abstraction cost at v1 is minimal.

**WhatsApp via Meta Cloud API instead of Twilio SMS**
Rejected. The user's primary messaging surface is native iOS Messages,
not WhatsApp. Twilio SMS delivers to the native app with no additional
app required. WhatsApp would require opening a secondary application.

**Telegram bot for inbound**
Rejected. Telegram requires a dedicated app on iOS. The friction of
opening a separate app to interact with Navi is inconsistent with
Principle 4 (reduce friction, never add it).

**Parsing inbound SMS with regex / keyword matching**
Rejected for the primary intent parser. Natural language input ("Katie
and I are seeing our interior designer next Tuesday, can you put that
on the calendar?") cannot be reliably parsed with keyword matching.
Claude API intent parsing handles natural language correctly. A
keyword pre-filter may be added as a performance optimisation but is
not the primary mechanism.
