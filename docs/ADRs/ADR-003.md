# ADR-003: CI/CD Pipeline and Environment Strategy

## Status
Accepted

## Date
2026-04-06

## Context

Navi requires a repeatable, automated deployment pipeline that provides
meaningful validation before code reaches production. The author's
preference is a three-tier environment model (dev, staging, prod)
connected by GitHub Actions, with automatic promotion from staging to
production on a tagged release, contingent on passing validation.

A fully isolated staging environment -- with its own dedicated Postgres
instance, NATS cluster, and Vault -- would be the architecturally
ideal approach. However, in a single-node homelab running personal
projects, the operational cost of maintaining a full parallel
infrastructure stack is disproportionate to the benefit. This ADR
documents a pragmatic staging model that provides meaningful risk
mitigation without doubling infrastructure burden.

## Decision

### Environment Model

Three environments are defined:

**dev**
Local development on the engineer's machine. Services run against a
local Postgres instance (Docker Compose), mocked or stubbed external
APIs, and no real credentials. Dev is never deployed to the homelab
node. The goal is fast iteration without any risk of affecting real
data or spending real money.

**staging**
Deployed to the homelab node, sharing physical infrastructure with
production. Isolation is achieved at the data and secrets layer:

- Dedicated Postgres schema: `navi_staging`
- Dedicated Vault paths: `secret/data/navi/staging/...`
- Sandbox/test API credentials for all paid external services:
  - Resend: sandbox mode (emails rendered but not delivered)
  - Twilio: test credentials (SMS not sent, no cost incurred)
  - Anthropic: real API key (no sandbox available; staging calls are
    real but low volume)
- NATS subjects prefixed with environment: `navi.staging.>`
- Distinct Docker container names to avoid conflicts with prod

Staging exists to validate that builds are deployable, migrations run
clean, and the system behaves correctly end-to-end -- without sending
real communications or touching production data.

**prod**
Deployed to the homelab node. Uses live credentials, the `navi_prod`
Postgres schema, and `navi.prod.>` NATS subjects. All real
communications (email, SMS) originate from prod only.

### Pipeline: Tagged Release Promotion

The deployment pipeline is triggered exclusively by a Git tag. Direct
pushes to main do not trigger a production deployment.

```
Developer creates tag (e.g. v1.2.0)
        |
        v
GitHub Actions: CI workflow
  - Run unit tests
  - Run linter (golangci-lint)
  - Build Docker image
  - Push image to registry tagged with version and 'staging'
        |
        v
GitHub Actions: Deploy to staging
  - Pull image tagged as 'staging'
  - Run database migrations against navi_staging schema
  - Start containers with staging compose profile
  - Run smoke test suite (see below)
        |
        v
Smoke tests pass?
  - FAIL: Pipeline stops. No production deployment.
           Notification sent via HA or fallback email.
  - PASS: Continue to production promotion.
        |
        v
GitHub Actions: Promote to prod
  - Re-tag image as 'prod' and 'latest'
  - Run database migrations against navi_prod schema
  - Rolling restart of prod containers
  - Run prod health checks
        |
        v
Notification: "Hey, listen! Navi v1.2.0 deployed successfully."
(delivered via SMS to confirm the delivery pipeline itself is working)
```

### Smoke Test Suite

Smoke tests run in staging after deployment and must all pass before
promotion. They are lightweight integration tests, not exhaustive unit
tests. Their purpose is to confirm the system is correctly wired, not
to validate business logic in depth.

Minimum smoke tests for v1:

| Test | Validates |
|------|-----------|
| Collector fetches at least one article from a known RSS feed | Collector is running and network is reachable |
| Summarizer returns a non-empty response from Claude API | Claude API key is valid and summarizer is wired |
| Delivery formats a digest without error | Digest template renders correctly |
| Resend sandbox call succeeds | Resend credentials are valid |
| Postgres migration status is clean | No pending migrations |
| NATS connection is established | Event bus is reachable |

Smoke tests are written as a dedicated Go test binary in
`services/digest/cmd/smoketest/` and run via a Makefile target:
`make smoketest ENV=staging`.

### Image Registry

Docker images are pushed to GitHub Container Registry (ghcr.io) using
the repository's built-in GitHub Actions credentials. Images are tagged
with the Git tag version, environment name, and a `latest` alias for
prod.

```
ghcr.io/{owner}/navi-digest:v1.2.0
ghcr.io/{owner}/navi-digest:staging
ghcr.io/{owner}/navi-digest:prod
ghcr.io/{owner}/navi-digest:latest
```

### Branch and Tag Conventions

```
main          -- stable, deployable at all times
feature/*     -- feature branches, merged via PR
fix/*         -- bug fix branches, merged via PR
v{major}.{minor}.{patch}  -- release tags, trigger the pipeline
```

No direct commits to main. All changes via pull request. Tags are
created manually by the author after reviewing what is on main.

### Makefile Targets

```
make dev              # Start local dev environment
make test             # Run unit tests
make lint             # Run linter
make migrate ENV=x    # Run migrations for environment x
make smoketest ENV=x  # Run smoke tests for environment x
make deploy ENV=x     # Manual deploy (bypasses pipeline, emergency use)
make logs ENV=x       # Tail logs for environment x
make status           # Show running containers across all environments
```

## Consequences

**Positive:**
- Tagged release model means production deployments are always
  intentional and version-tracked.
- Smoke tests catch wiring errors (bad credentials, missing env vars,
  broken migrations) before they reach prod.
- Sandbox API keys in staging mean no real money is spent and no real
  communications are sent during validation.
- Schema isolation means a bad migration in staging cannot corrupt
  production data.

**Negative / tradeoffs:**
- Staging shares physical infrastructure with production. A
  catastrophic node failure takes down both simultaneously. Accepted
  for a single-node homelab.
- Anthropic has no sandbox mode. Staging summarizer calls are real API
  calls. Volume is low (smoke tests only) and cost is negligible, but
  this is worth noting.
- The promotion pipeline assumes the homelab node is reachable from
  GitHub Actions runners. This requires either a public ingress or a
  self-hosted GitHub Actions runner on the node. A self-hosted runner
  is the preferred approach to avoid exposing additional ports.

## Alternatives Considered

**Deploy on every push to main**
Rejected. Continuous deployment to production without a validation gate
is too aggressive for a system that sends real SMS and email. A bad
push would result in failed deliveries or unexpected communications.

**Manual deployments only**
Rejected. Manual deployments are error-prone and create drift between
what is tagged and what is running. The tagged release pipeline
provides automation without sacrificing intentionality.

**Full isolated staging environment**
Rejected. A dedicated Postgres instance, NATS cluster, and Vault for
staging would double the operational surface area of the homelab with
no meaningful additional protection for a single-user personal project.
Schema and secret isolation on shared infrastructure is sufficient.
