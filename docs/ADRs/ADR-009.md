# ADR-009: Monitoring and Alerting

## Status
Accepted

## Date
2026-04-06

## Context

Navi runs unattended. The daily digest is the most visible indicator
of system health -- if it doesn't arrive, something is wrong -- but
by the time a missing digest is noticed, the failure may be hours old.
Proactive monitoring is required to surface problems before they
manifest as a missing delivery.

The Foundation infrastructure already includes Grafana and Prometheus.
Uptime Kuma is used for uptime monitoring. Navi's monitoring must
integrate with this existing stack rather than introduce parallel
tooling.

The on-call model is simple: there is one user and one operator, and
they are the same person. Alerting must be actionable and low-noise --
a system that pages frequently is a system that gets ignored.

## Decision

### Prometheus Integration

Navi metrics (defined in ADR-008) are scraped by the Foundation
Prometheus instance. The OTEL Collector exposes a Prometheus-compatible
metrics endpoint on port 9090, which is added to the Foundation
Prometheus scrape configuration.

```yaml
# Foundation prometheus.yml addition
scrape_configs:
  - job_name: navi
    static_configs:
      - targets: ['10.0.40.10:9090']
    relabel_configs:
      - source_labels: [__address__]
        target_label: instance
        replacement: navi
```

No new Prometheus instance is deployed for Navi.

### Jaeger for Distributed Tracing

A Jaeger instance is deployed as part of the Navi Docker Compose
stack. Traces from the OTEL Collector are exported to Jaeger via
OTLP. Jaeger's UI is accessible via Traefik on the internal network.

Jaeger uses in-memory storage for v1 with a retention of 72 hours.
This is sufficient for debugging recent failures without adding a
persistent storage dependency. If trace retention beyond 72 hours
becomes useful, Jaeger can be configured with Badger persistent
storage at that time.

### Grafana Dashboards

A Navi dashboard is added to the Foundation Grafana instance. The
dashboard is defined as a JSON provisioning file committed to the
navi repository under `monitoring/grafana/`. It is loaded
automatically by Grafana on startup via a provisioned data source.

The v1 dashboard contains the following panels:

**System Health**
- Service uptime (liveness probe history)
- NATS connection status
- Postgres connection status

**Collection**
- Articles fetched per feed per day (bar chart)
- Feed error rate by feed (time series)
- Deduplication rate (gauge -- high dedup rate signals feed issues)
- Feed poll latency by tier (heatmap)

**Digest Generation**
- Digests generated per type per week (stat panels)
- Digest delivery status (delivered vs. failed, last 30 days)
- Articles per digest by bucket (bar chart)
- Relevance score distribution (histogram)

**API Costs**
- Claude API spend per day (time series)
- Claude API spend cumulative this month (stat panel with threshold)
- API call volume by operation (bar chart)
- Claude API error rate (time series)

**Delivery**
- Last digest delivered (stat panel -- turns red if > 25 hours)
- Email delivery success rate
- SMS delivery success rate
- Delivery latency by channel

### Uptime Kuma

The Navi health endpoints (`/healthz/live`, `/healthz/ready`) are
added to the Foundation Uptime Kuma instance for uptime monitoring.
Uptime Kuma polls these endpoints every 60 seconds and alerts if
they return non-200 or fail to respond within 10 seconds.

### Alert Conditions

Alerts are defined in Prometheus alert rules and routed to the user
via SMS (Twilio) using the Alertmanager Webhook receiver. Alerts
are sent from the Navi system number so they arrive in the same
thread as other Navi messages, prefixed with "Hey, listen!"

Alert noise is taken seriously. Every alert must be actionable --
if the correct response to an alert is "ignore it", the alert should
not exist.

**Critical alerts -- immediate SMS:**

```
DigestNotDelivered
  expr: time() - navi_last_digest_delivered_timestamp > 90000
  for: 5m
  summary: Daily digest has not been delivered in over 25 hours.
  -- Fires if the morning digest fails entirely.

NaviServiceDown
  expr: up{job="navi"} == 0
  for: 2m
  summary: Navi service is unreachable.
  -- Fires if health endpoint stops responding.

DBConnectionFailed
  expr: navi_db_errors_total{error_type="connection"} > 0
  for: 1m
  summary: Navi cannot reach Postgres.

ClaudeAPIBudgetExceeded
  expr: navi_claude_api_cost_usd_total > 10
  for: 0m
  summary: Claude API spend has exceeded $10 this month.
  -- Threshold is generous for personal use; catches runaway loops.
```

**Warning alerts -- SMS during waking hours only (7am-10pm):**

```
FeedErrorRateHigh
  expr: rate(navi_feed_poll_errors_total[1h]) > 0.2
  for: 30m
  summary: More than 20% of feed polls are failing.
  -- Fires if multiple feeds are broken, not just one.

DeliveryErrorRateHigh
  expr: rate(navi_delivery_errors_total[1h]) > 0
  for: 15m
  summary: Delivery errors detected on one or more channels.

ClaudeAPIErrorRateHigh
  expr: rate(navi_claude_api_errors_total[5m]) > 0.1
  for: 10m
  summary: Claude API error rate exceeds 10%.
```

**Informational -- no alert, visible in dashboard only:**
- Individual feed timeouts (single feed, single occurrence)
- Deduplication rate spikes (expected behavior, not a failure)
- Digest article count below average (signal quality, not system
  health)

### Cost Guardrails

Beyond the ClaudeAPIBudgetExceeded alert, hard circuit breaker
logic is implemented in the summarizer:

- If Claude API spend in the current calendar month exceeds a
  configured monthly limit (default $15, stored in Vault at
  `secret/data/navi/{env}/limits/claude_monthly_usd`), the
  summarizer stops making API calls and queues a warning digest
  containing only the raw article headlines without summaries.
- The circuit breaker resets on the 1st of the month.
- The limit is configurable via Vault without a redeployment.

This ensures that a bug causing repeated summarizer calls cannot
generate unbounded spend. The digest degrades gracefully rather
than silently failing.

### Runbook

A `monitoring/RUNBOOK.md` file is committed to the navi repository
documenting the expected response to each alert condition. For a
single-operator system this is as much a memory aid as a procedure --
it ensures that the reasoning behind each alert is preserved even
if months pass between incidents.

## Consequences

**Positive:**
- Integrating with Foundation's existing Grafana and Prometheus
  avoids operating a separate monitoring stack for Navi.
- SMS delivery of alerts means actionable failures surface in the
  same channel used for normal Navi interaction.
- The cost guardrail circuit breaker means a runaway bug degrades
  gracefully rather than running up an unexpected bill.
- Dashboard provisioned from the repository means monitoring
  configuration is versioned and portable.

**Negative / tradeoffs:**
- Alertmanager must be deployed and configured in Foundation if it
  is not already present. This is a small one-time setup cost.
- Jaeger in-memory storage means traces are lost on container
  restart. Accepted for v1; persistent storage is a future option.

## Alternatives Considered

**PagerDuty or similar on-call tooling**
Rejected. The on-call model is one person. Twilio SMS via
Alertmanager webhook is sufficient and avoids a paid external
dependency.

**Email-only alerts**
Rejected. Email is the digest delivery channel. Mixing operational
alerts into the same inbox as digest content makes both less useful.
SMS is the right channel for alerts that require prompt attention.

**No alerting, rely on noticing missing digests**
Rejected. A missing digest is a lagging indicator. The feed error
rate and service health alerts are leading indicators that surface
problems before they cause a missed delivery.
