# ADR-004: Core Data Model

## Status
Accepted

## Date
2026-04-06

## Context

Navi's feature set -- news digest, trend analysis, Rolodex, professional
profile, meeting prep, research briefs -- shares a common underlying
structure: named entities (people, companies, topics) connected to
content (articles, digests, notes) over time. The data model must
support this from day one, even though only the digest feature is being
built in v1.

The decisions made here are the most expensive to change later. A
schema designed only for the digest feature will fight every subsequent
feature. A schema designed for the full knowledge graph will serve v1
just as well while leaving every door open.

Two storage capabilities are required:
- Relational storage for structured entities and their relationships
- Semantic storage for similarity search across unstructured content

Both are available in the Foundation Postgres instance via standard
Postgres and the pgvector extension respectively.

## Decision

### Entity Model

Navi's data model is a knowledge graph represented relationally. The
core entities are:

```
articles         -- raw fetched content from external sources
digests          -- generated summaries delivered to the user
people           -- contacts in the Rolodex
companies        -- organisations tracked by Navi
topics           -- named themes (e.g. "BNPL", "Capital One", "Fed rates")
events           -- calendar events (future)
interactions     -- logged contact with a person (future)
profile          -- the user's professional identity (single row)
```

Entities are linked via join tables that carry metadata about the
relationship:

```
article_topics        -- an article covers a topic
article_companies     -- an article mentions a company
article_people        -- an article mentions a person
digest_articles       -- an article was included in a digest
person_companies      -- a person works at / worked at a company
person_topics         -- a person is relevant to a topic
company_topics        -- a company is relevant to a topic
```

This structure means that when a new article is collected and enriched,
it is linked to the relevant people, companies, and topics already in
the graph. Meeting prep, competitive intelligence, and relationship
nudges are then queries across these join tables -- no new schema
required.

### Schema Definition (v1 tables)

The following tables are required for the digest service (v1). All
tables live in the `navi_{env}` schema.

```sql
-- Content sources
CREATE TABLE feeds (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name        TEXT NOT NULL,
    url         TEXT NOT NULL UNIQUE,
    bucket      TEXT NOT NULL,       -- 'tech_banking' | 'global_events'
    tier        TEXT NOT NULL,       -- 'wire' | 'daily' | 'analytical'
    enabled     BOOLEAN NOT NULL DEFAULT true,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Raw fetched articles
CREATE TABLE articles (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    feed_id         UUID NOT NULL REFERENCES feeds(id),
    url             TEXT NOT NULL UNIQUE,
    title           TEXT NOT NULL,
    raw_content     TEXT,
    published_at    TIMESTAMPTZ,
    collected_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    embedding       vector(1536),    -- pgvector; populated by enricher
    relevance_score FLOAT,           -- scored by summarizer
    CONSTRAINT articles_url_unique UNIQUE (url)
);

-- Generated digests
CREATE TABLE digests (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    digest_type     TEXT NOT NULL,   -- 'daily' | 'weekly' | 'monthly'
    period_start    TIMESTAMPTZ NOT NULL,
    period_end      TIMESTAMPTZ NOT NULL,
    content_html    TEXT NOT NULL,
    content_text    TEXT NOT NULL,
    delivered_at    TIMESTAMPTZ,
    opened_at       TIMESTAMPTZ,     -- future: read tracking
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Which articles were included in which digest
CREATE TABLE digest_articles (
    digest_id   UUID NOT NULL REFERENCES digests(id),
    article_id  UUID NOT NULL REFERENCES articles(id),
    section     TEXT NOT NULL,       -- 'tech_banking' | 'global_events'
    summary     TEXT NOT NULL,       -- the generated per-article summary
    PRIMARY KEY (digest_id, article_id)
);

-- Named topics (seed data + enricher-created)
CREATE TABLE topics (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name        TEXT NOT NULL UNIQUE,
    slug        TEXT NOT NULL UNIQUE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Article-topic links
CREATE TABLE article_topics (
    article_id  UUID NOT NULL REFERENCES articles(id),
    topic_id    UUID NOT NULL REFERENCES topics(id),
    confidence  FLOAT NOT NULL DEFAULT 1.0,
    PRIMARY KEY (article_id, topic_id)
);
```

The following tables are defined now but populated in later features.
Defining them early ensures migrations are additive rather than
destructive.

```sql
-- People (Rolodex)
CREATE TABLE people (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            TEXT NOT NULL,
    email           TEXT,
    phone           TEXT,
    notes           TEXT,
    last_contacted  TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Companies
CREATE TABLE companies (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name        TEXT NOT NULL UNIQUE,
    domain      TEXT,
    industry    TEXT,
    notes       TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Person-company relationships
CREATE TABLE person_companies (
    person_id   UUID NOT NULL REFERENCES people(id),
    company_id  UUID NOT NULL REFERENCES companies(id),
    role        TEXT,
    current     BOOLEAN NOT NULL DEFAULT true,
    started_at  DATE,
    ended_at    DATE,
    PRIMARY KEY (person_id, company_id)
);

-- Article-company links
CREATE TABLE article_companies (
    article_id  UUID NOT NULL REFERENCES articles(id),
    company_id  UUID NOT NULL REFERENCES companies(id),
    confidence  FLOAT NOT NULL DEFAULT 1.0,
    PRIMARY KEY (article_id, company_id)
);

-- Article-person links
CREATE TABLE article_people (
    article_id  UUID NOT NULL REFERENCES articles(id),
    person_id   UUID NOT NULL REFERENCES people(id),
    confidence  FLOAT NOT NULL DEFAULT 1.0,
    PRIMARY KEY (article_id, person_id)
);

-- Professional profile (single row, owned by the user)
CREATE TABLE profile (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title           TEXT,
    company         TEXT,
    domain          TEXT,
    tenure_start    DATE,
    bio             TEXT,           -- positioning narrative
    goals           TEXT,           -- developmental goals (free text, updated via SMS)
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Portfolio entries (linked to profile)
CREATE TABLE portfolio_entries (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title       TEXT NOT NULL,
    description TEXT NOT NULL,
    outcomes    TEXT,
    period      TEXT,               -- e.g. "Q1 2025"
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### Semantic Layer

pgvector is used for embedding storage and similarity search.
The `embedding` column on the `articles` table stores a 1536-dimension
vector (compatible with OpenAI text-embedding-3-small and similar
models). Embeddings are generated by the enricher component after
collection and are used for:

- **Deduplication**: articles with cosine similarity above a threshold
  (0.92) against recently collected articles are discarded
- **Trend analysis**: clustering articles by embedding similarity
  to surface themes across a weekly or monthly window
- **Research retrieval**: semantic search across the article corpus
  when generating on-demand research briefs (Later horizon)

An IVFFlat index is created on the embedding column once the article
corpus exceeds ~10,000 rows. For smaller corpora, a sequential scan
is sufficient and an index adds overhead without benefit.

```sql
-- Created when corpus is large enough
CREATE INDEX articles_embedding_idx
    ON articles
    USING ivfflat (embedding vector_cosine_ops)
    WITH (lists = 100);
```

### Preference and Feedback Model

Navi improves over time by tracking what the user engages with. A
lightweight signal table captures read events and explicit feedback
without requiring any active input beyond natural interaction.

```sql
CREATE TABLE feedback_signals (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    entity_type TEXT NOT NULL,      -- 'digest' | 'article' | 'topic'
    entity_id   UUID NOT NULL,
    signal      TEXT NOT NULL,      -- 'opened' | 'clicked' | 'thumbs_up' | 'thumbs_down'
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

These signals are aggregated by the summarizer to weight topic
relevance and source quality over time.

## Consequences

**Positive:**
- The entity model supports every feature in the roadmap without
  structural changes. New features add tables and join tables; they
  do not restructure existing ones.
- Defining people, companies, and portfolio tables in v1 migrations
  means the schema is always ahead of the features, avoiding
  disruptive schema changes later.
- pgvector on Postgres keeps the stack simple -- no separate vector
  database to operate.

**Negative / tradeoffs:**
- The schema is more complex than strictly required for a news digest
  alone. This is intentional and accepted in exchange for long-term
  stability.
- Embedding generation adds latency and API cost to the enrichment
  pipeline. For v1 deduplication, a simpler URL-based dedup is
  sufficient and embeddings can be generated asynchronously.

## Alternatives Considered

**Design only the articles and digests tables for v1**
Rejected. The cost of retrofitting a knowledge graph schema onto an
articles-only schema is high. Adding the entity tables now costs
nothing -- they are empty tables until features use them.

**A dedicated graph database (Neo4j, etc.)**
Rejected. The relationships in this data model are structured and
finite. Postgres with foreign keys and join tables handles them well.
A graph database adds significant operational overhead for queries
that relational joins can answer adequately.

**A separate vector database (Pinecone, Qdrant, etc.)**
Rejected. pgvector on the existing Foundation Postgres instance is
sufficient for the corpus sizes Navi will encounter. Introducing a
separate vector database would add an operational dependency with no
benefit at this scale.
