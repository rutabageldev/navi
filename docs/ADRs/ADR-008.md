# ADR-008: Observability and Telemetry

## Status
Accepted

## Date
2026-04-06

## Context

Navi is a multi-component system: collector, enricher, summarizer,
scheduler, intent handler, and delivery dispatcher. These components
communicate across NATS subjects and call multiple external APIs.
Without structured observability, diagnosing failures, latency issues,
and unexpected behavior requires guesswork.

Observability in Navi has three pillars: traces (what happened and
how long it took), metrics (aggregate system health over time), and
logs (discrete events with context). All three must be correlated by
a common trace ID so a single digest delivery can be followed from
RSS fetch through Claude API call through Resend delivery without
reconstructing a timeline manually.

The existing Foundation infrastructure includes Grafana and Prometheus.
Navi's observability stack must integrate with these rather than
introduce parallel tooling.

## Decision

### OpenTelemetry as the Instrumentation Standard

All Navi services are instrumented using the OpenTelemetry Go SDK.
OTEL is the instrumentation standard -- no service-specific metrics
libraries, no direct Prometheus client instrumentation in application
code. The OTEL SDK emits to a collector, and the collector is
responsible for exporting to Prometheus and any other backend.

The OTEL Collector runs as a sidecar container in the Navi Docker
Compose stack. It receives telemetry from Navi services via OTLP
(gRPC on port 4317) and exports:
- Metrics to the Foundation Prometheus instance
- Traces to a Jaeger instance (deployed in Foundation or as a Navi
  sidecar -- see Monitoring ADR-009)

```
Navi service --> OTLP/gRPC --> OTEL Collector --> Prometheus (metrics)
                                               --> Jaeger (traces)
```

### Trace Propagation

Every operation that crosses a service or component boundary carries
a trace context. This includes:

- Inbound HTTP requests (Twilio webhook, health endpoints)
- Outbound HTTP requests (Claude API, Resend, Twilio, RSS feeds)
- NATS message publish and subscribe

NATS does not natively support W3C Trace Context headers. Trace
context is propagated by embedding the traceparent and tracestate
values in the CloudEvents envelope under the `traceparent` and
`tracestate` extension attributes. Every NATS publisher extracts
the current span context and writes it into the outgoing envelope.
Every NATS subscriber extracts the span context from the envelope
and starts a child span before processing.

This ensures that a single article collected by the collector
produces a continuous trace through enrichment, summarization, and
delivery -- visible as a single trace in Jaeger with child spans
per component.

### Span Naming Conventions

Spans are named using the pattern `{component}.{operation}`:

```
collector.feed_poll
collector.article_fetch
enricher.entity_extract
enricher.embedding_generate
summarizer.relevance_score
summarizer.article_summarize
summarizer.digest_generate
delivery.email_render
delivery.email_send
delivery.sms_send
intent.parse
intent.handle
store.query
store.insert
```

All spans include a standard set of attributes:

```
navi.environment        -- dev | staging | prod
navi.service            -- digest | intent | delivery
navi.component          -- collector | enricher | summarizer | etc.
navi.feed.name          -- (collector spans) feed name
navi.digest.type        -- (summarizer spans) daily | weekly | monthly
navi.intent.type        -- (intent spans) detected intent
```

External API calls use the standard OTEL semantic conventions for
HTTP client spans, augmented with:

```
navi.external.service   -- anthropic | resend | twilio | rss
navi.external.cost_usd  -- (Claude API spans) estimated cost of call
```

### Metrics

The following metrics are defined for v1. All metrics are prefixed
with `navi_`.

**Collector:**
```
navi_articles_fetched_total         counter  {feed, bucket}
navi_articles_deduplicated_total    counter  {reason: url|semantic}
navi_feed_poll_duration_seconds     histogram {feed, tier}
navi_feed_poll_errors_total         counter  {feed, error_type}
```

**Summarizer:**
```
navi_digests_generated_total        counter  {type: daily|weekly|monthly}
navi_articles_scored_total          counter  {bucket, above_threshold}
navi_claude_api_calls_total         counter  {operation, model}
navi_claude_api_duration_seconds    histogram {operation}
navi_claude_api_cost_usd_total      counter  {operation}
navi_claude_api_errors_total        counter  {operation, error_type}
```

**Delivery:**
```
navi_digests_delivered_total        counter  {channel, type, status}
navi_delivery_duration_seconds      histogram {channel}
navi_delivery_errors_total          counter  {channel, error_type}
```

**Intent:**
```
navi_sms_received_total             counter  {authorized: true|false}
navi_intents_parsed_total           counter  {intent_type}
navi_intents_handled_total          counter  {intent_type, status}
navi_intent_parse_duration_seconds  histogram
```

**Store:**
```
navi_db_query_duration_seconds      histogram {operation, table}
navi_db_errors_total                counter  {operation, error_type}
```

### Structured Logging

All Navi services use structured logging via the Go `slog` package
(standard library, Go 1.21+). All log output is JSON. No unstructured
log.Printf statements anywhere in application code.

Every log entry includes the following base fields:

```json
{
  "time":        "2026-04-06T06:30:00Z",
  "level":       "INFO",
  "service":     "digest",
  "component":   "collector",
  "environment": "prod",
  "trace_id":    "4bf92f3577b34da6a3ce929d0e0e4736",
  "span_id":     "00f067aa0ba902b7",
  "msg":         "article fetched",
  "feed":        "Reuters Technology",
  "url":         "https://..."
}
```

The `trace_id` and `span_id` fields are populated from the active
OTEL span context. This means every log line is directly correlatable
with the corresponding trace in Jaeger.

Log levels are used consistently:

```
DEBUG   -- high-frequency operational detail, disabled in prod by default
INFO    -- normal operational events (article fetched, digest sent)
WARN    -- unexpected but recoverable conditions (feed timeout, retry)
ERROR   -- failures requiring attention (digest not delivered, API down)
```

The log level is configured per environment via an environment
variable (`NAVI_LOG_LEVEL`) with a default of INFO in prod and
DEBUG in dev.

### Health Endpoints

Every Navi service exposes two HTTP endpoints:

```
GET /healthz/live    -- liveness: is the process running?
GET /healthz/ready   -- readiness: is the service ready to handle work?
```

The liveness endpoint returns 200 always if the process is running.
The readiness endpoint checks:
- Postgres connection is reachable
- NATS connection is established
- Vault is reachable (at startup only -- credentials are cached)

These endpoints are used by Docker health checks and by the smoke
test suite (ADR-003). They are not authenticated -- they return
status only, no sensitive information.

### Error Tracking

All ERROR level log events are also published to the
`navi.{env}.errors` NATS subject as a CloudEvents envelope. This
allows a future error aggregation subscriber to collect, deduplicate,
and alert on errors without coupling the alerting logic into each
service. The error event schema is:

```json
{
  "specversion": "1.0",
  "type":        "navi.error.reported",
  "source":      "navi/digest/collector",
  "id":          "{uuid}",
  "time":        "2026-04-06T06:30:00Z",
  "data": {
    "level":     "ERROR",
    "service":   "digest",
    "component": "collector",
    "message":   "feed fetch failed after 3 retries",
    "feed":      "Reuters Technology",
    "trace_id":  "4bf92f3577b34da6a3ce929d0e0e4736"
  }
}
```

## Consequences

**Positive:**
- A single trace ID correlates log lines, spans, and metrics for any
  given operation. Debugging a failed digest delivery is a Jaeger
  query, not a log grep.
- OTEL Collector as the central export point means the backend can
  change (swap Jaeger for Tempo, add a second metrics sink) without
  touching application code.
- Structured logs with trace correlation make the system observable
  from day one, not bolted on after the first production incident.
- navi_claude_api_cost_usd_total enables the cost guardrail alerting
  defined in ADR-009.

**Negative / tradeoffs:**
- OTEL instrumentation adds boilerplate to every component. A shared
  internal library (`internal/telemetry`) wraps the SDK initialization
  and provides helpers for NATS context propagation to reduce per-
  component overhead.
- The OTEL Collector is an additional container to operate. It is
  lightweight but adds to the service count.

## Alternatives Considered

**Direct Prometheus instrumentation without OTEL**
Rejected. Direct Prometheus client instrumentation creates a tight
coupling between application code and the metrics backend. OTEL
provides backend agnosticism and adds distributed tracing that
Prometheus alone cannot provide.

**Unstructured logging with log aggregation**
Rejected. Unstructured logs cannot be reliably correlated with traces
or queried programmatically. slog with JSON output is no more complex
to write than fmt.Printf and is significantly more useful to read.

**A separate logging service (Loki, ELK)**
Deferred. stdout JSON logs collected by Docker's log driver are
sufficient for v1. If log volume grows to the point where Grafana
Loki becomes valuable, it can be added to Foundation without any
changes to Navi's logging code -- the format is already compatible.
