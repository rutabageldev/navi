# Session — 2026-04-10

## What was done

This session completed the CI/CD pipeline established in the previous session and achieved the first successful end-to-end automated deployment to production (v0.1.4).

**CI/CD fixes (5 PRs, 5 releases):**

- **PR #3** (`fix/release-please-action`): Switched from deprecated `google-github-actions/release-please-action@v4` to the canonical `googleapis/release-please-action@v4`. The deprecated action was silently ignoring `bump-minor-pre-major: true` and the `.versions.json` manifest, proposing `1.0.0` instead of `0.1.0`. → Cut `v0.1.0` (deploy did not fire — see below).

- **PR #5** (`fix/release-please-version`): Moved `release-type`/`bump-minor-pre-major` inside `packages["."]` (global-level options were ignored for the `go` release type) and added a one-time `release-as: "0.1.0"` override to force the correct initial version. → Cut `v0.1.0` correctly.

- **PR #7** (`fix/deploy-workflow-dispatch`): Two fixes bundled:
  1. `GITHUB_TOKEN`-created tags do not trigger downstream workflows (GitHub security restriction). Switched `release-please.yml` to use `RELEASE_PLEASE_TOKEN` (fine-grained PAT: contents + pull-requests read/write) so tag pushes propagate to `deploy.yml`.
  2. Added `workflow_dispatch` to `deploy.yml` as a manual fallback, with dual-source version resolution (`workflow_dispatch` input vs `GITHUB_REF` tag).
  3. Removed `release-as: "0.1.0"` override now that the first release was cut.
  → Cut `v0.1.1`; deploy fired but failed (VAULT_TOKEN not found).

- **PR #9** (`fix/deploy-load-env`): The deploy workflow checks out to the Actions workspace, not `/opt/navi/`. `deploy.sh` looks for `.env` relative to itself and failed. Fixed by loading `/opt/navi/.env` into `$GITHUB_ENV` as the first step of the deploy job. → Cut `v0.1.2`; deploy reached prod but failed on health check (wrong port — see below).

- **PR #11** (`fix/deploy-pipeline-and-port`): Two fixes:
  1. Port conflict: `navi-digest-prod` was binding port `8083`, which is permanently owned by Traefik's metrics entrypoint in Foundation (`traefik.yml` line 14). Changed to `8084`. Updated `docker-compose.yml` and `smoketest/main.go`.
  2. Restructured `deploy.yml` from one monolithic job into five discrete pipeline jobs: `build → deploy-staging → test-staging → deploy-prod → test-prod`. State (version, built, prev) flows via job outputs. Rollback logic stays co-located with its test job.
  → Cut `v0.1.3`; deploy fired, all jobs passed except `test-prod` (health check still hitting 8083 — `service-addr.sh` was missed).

- **PR #13** (`fix/prod-port-service-addr`): Updated `service-addr.sh` prod address from `8083` to `8084` (missed in PR #11). Also renamed containers to match `ruby-core` convention: `{product}-{env}-{service}` → `navi-prod-digest`, `navi-staging-digest`, `navi-dev-digest`. → Cut `v0.1.4`; **full pipeline green** for the first time.

- **PR #15** (`chore/docker-ps-target`): Replaced `make status` with `make docker-ps`, matching Foundation's aligned column format (`column -t -s "$(printf '\t')"`).

The v0.1.3 deploy did trigger a rollback attempt, but it failed because `.last-deployed-version` was `none` (first deploy). A proper drill — deploy a bad service version, confirm staging rollback fires and SMS is received, confirm prod was not touched — has not been done.

2. **SMS delivery:** The `notify-sms.sh` script is wired and will call Twilio, but Twilio credentials have not been seeded in Vault. All SMS steps currently exit 0 silently (Vault unreachable fallback). This is a P1 dependency; the P0 exit criterion for SMS is blocked on Twilio setup.

3. **Observability verification:** Three P0 exit criteria remain unverified:
   - At least one trace visible in Foundation Tempo (from a prod health check)
   - Tempo → Loki trace_id navigation working in both directions
   - At least one JSON log queryable in Loki: `{container_name="navi-prod-digest"} | json`
   These require manual spot-checking in Grafana now that prod is running.

4. **Uptime Kuma:** Prod health endpoints (`/v1/health/live` and `/v1/health/ready` at `10.0.40.10:8084`) have not been added to Foundation Uptime Kuma under a Navi group. Required for P0 completion.

5. **Service version in health response:** The plan requires that the version field in `/v1/health/ready` matches the deployed tag. This has not been spot-checked against the live prod container.

6. **Node.js 20 deprecation:** All GitHub Actions steps emit a warning about Node.js 20 actions being deprecated (forced to Node.js 24 in June 2026). `actions/checkout@v4`, `docker/login-action@v3`, `actions/setup-go@v5` all have v5/v4 updates available. Not blocking but should be addressed before the June deadline.
